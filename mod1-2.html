The DevOps Lifecycle

The DevOps lifecycle is an infinite loop of continuous improvement:

12/40

DevOps Lifecycle: 8 Phases

  • Plan: Define requirements, track work, prioritize features (project management tools)
  • Code: Write and version control your code
  • Build: Compile code, manage dependencies, create artifacts
  • Test: Automated testing to catch issues early
  • Release: Prepare for deployment, manage release schedules
  • Deploy: Get your code into production
  • Operate: Keep systems running, manage infrastructure
  • Monitor: Track performance, logs, user experience
13/40

Benefits of DevOps

  • Faster Delivery: Deploy multiple times per day instead of waiting days or weeks
  • Improved Collaboration: Less blame, more teamwork between Dev and Ops
  • Greater Reliability: Automation reduces human error and ensures consistent environments
  • Reduced Stress: Fewer 3 AM pages, fewer deployment nightmares
  • Complete Developer: Understanding the full software delivery lifecycle makes you more effective

Permissions (chmod, chown)

# Understanding permissions # -rwxr-xr-- means: # Owner: read, write, execute (rwx) # Group: read, execute (r-x) # Others: read only (r--) # Changing permissions chmod +x script.sh # Make script executable chmod 755 file.txt # Owner: rwx, Group: rx, Others: rx chmod 644 config.yml # Owner: rw, Group: r, Others: r # Changing ownership chown user:group file.txt # Change owner and group sudo chown root:root app # Change to root (needs sudo)

Introduction to Docker

Docker Image

  • Contains app + dependencies + OS libraries
  • Immutable (doesn't change)
  • Can be stored and shared

Docker Container

  • Lightweight (shares host OS kernel)
  • Isolated (can't interfere with others)
  • Disposable (stop and start easily)
Think of it this way: Image = Recipe | Container = Meal cooked from that recipe

Container Best Practices

  • Keep images small: Use slim/alpine base images (200MB vs 500MB+)
  • One process per container: Don't run app + database + nginx together
  • Don't run as root: Create a non-root user for security
  • Use .dockerignore: Like .gitignore, prevents copying unnecessary files
  • Version your images: Never use latest in production - use v1.2.3
RUN useradd -m appuser USER appuser
35/40

Container Security

Security concerns to watch for:
  • Vulnerable base images: Scan images for known vulnerabilities
  • Secrets in images: Never put passwords in Dockerfiles
  • Privileged containers: Avoid running with elevated privileges
  • Outdated images: Regularly rebuild with updated base images
Best practice: Use environment variables or secret management services for sensitive data
